\begin{abstract}
Open-vocabulary detectors can localize language-relevant objects in 2D, but
robots require 3D action targets that are geometrically consistent, associated
across views, and executable by a controller. This paper studies that
retrieval-to-execution gap. We present Query-to-Grasp, a reproducible
diagnostic RGB-D manipulation framework that connects GroundingDINO proposals,
optional CLIP reranking, RGB-D lifting, multi-view object memory,
confidence-aware re-observation, and simulated pick and pick-place evaluation
in ManiSkill. The evaluation separates semantic retrieval from downstream
control by reporting \texttt{pick\_success}, \texttt{place\_success}, and
\texttt{task\_success}. CLIP reranking did not change the top-1 candidate in
our evaluated low-candidate scenes, while noisy-oracle probes show that 2--5 cm
target perturbations can collapse pick or place success. After camera-frame
alignment, fused memory grasp points achieve \texttt{1.0000} simulated pick
success on \texttt{PickCube-v1}. On \texttt{StackCube-v1}, the system separates
pick-only transfer, privileged oracle upper bounds, query-pick/oracle-place
diagnostic bridges, and a non-oracle explicit reference-object bridge. The
non-oracle bridge reaches \texttt{0.552}, \texttt{0.472}, and \texttt{0.446}
task success over 500 seeds in single-view, tabletop, and closed-loop modes,
showing measurable but incomplete closure of the placement-target gap. These results position
Query-to-Grasp as a target-source diagnostic benchmark for studying how
open-vocabulary RGB-D retrieval becomes, or fails to become, executable
simulated manipulation.
\end{abstract}

\begin{IEEEkeywords}
Open-vocabulary manipulation, RGB-D perception, multi-view object memory,

\section{Introduction}

Open-vocabulary detectors can localize language-described objects in images,
but robotic manipulation requires a stronger interface: the selected object
must become a spatial target that can survive RGB-D lifting, cross-view fusion,
and contact-rich execution. This intermediate layer is often hidden inside
larger manipulation systems. Query-to-Grasp isolates it as the object of study.
The central question is not only whether a detector finds the right pixels, but
whether the resulting target source is geometrically stable and physically
usable.

The system parses a natural-language query, obtains 2D proposals with
GroundingDINO~\cite{liu2023groundingdino}, optionally reranks candidates with
CLIP~\cite{radford2021learning}, lifts RGB-D observations into world
coordinates, fuses multi-view detections into object memory, and selects a
query-matching 3D target. A controlled retrieval-only baseline measures
perception and uncertainty without conflating them with control, while
simulated pick and pick-place executors test whether selected targets can drive
ManiSkill actions.

The paper is organized around four diagnostic observations. First, reranking is
not the main bottleneck in the current candidate regime: optional CLIP
reranking rarely changes the top selection, so failures are better explained by
3D target quality. Second, camera-frame consistency is a prerequisite for
multi-view object memory: once RGB-D lifting and simulator camera poses are
aligned, cross-view spread and memory fragmentation drop sharply. Third,
\texttt{PickCube-v1} demonstrates executability: fused grasp targets carried
through object memory reach \texttt{1.0000} simulated pick success. Fourth,
\texttt{StackCube-v1} exposes the remaining retrieval-to-execution gap through
pick-only transfer, predicted reference-object placement, and privileged
oracle upper-bound probes.

This paper makes three contributions:
\begin{itemize}
\item A target-source diagnosis protocol that explicitly separates semantic
centers, refined and memory grasp points, task-guarded targets, oracle targets,
and predicted reference-object targets.
\item Evidence that camera-frame alignment and grasp-point memory are the key
interfaces between open-vocabulary 2D grounding and executable simulated
picking.
\item A PickCube/StackCube diagnostic evaluation showing that semantic
retrieval success is not manipulation success: failures concentrate in 3D
target quality, cross-view association, and place-target precision.
\end{itemize}

No real-robot execution is claimed. The controller is scripted, not learned,
and StackCube results do not claim robust relation-heavy language-conditioned
stacking. The contribution is a benchmarkable diagnostic system for
target-source quality, cross-view geometric consistency, and re-observation
association in language-conditioned RGB-D manipulation.

\section{Related Work}

\subsection{Open-Vocabulary 2D and 3D Scene Grounding}

Open-vocabulary vision-language models provide reusable perception primitives
for language-conditioned robotics. CLIP~\cite{radford2021learning} aligns
image and text embeddings, while GroundingDINO~\cite{liu2023groundingdino}
grounds language prompts into 2D detections. Recent 3D grounding and scene
mapping systems embed language-aligned features into point clouds, fields, or
maps, including OpenScene~\cite{peng2023openscene},
ConceptFusion~\cite{jatavallabhula2023conceptfusion},
CLIP-Fields~\cite{shafiullah2023clipfields}, and
LERF~\cite{kerr2023lerf}. Query-to-Grasp is complementary: it does not propose
a new 2D or 3D representation, but tests whether the target source produced
from RGB-D grounding can drive executable simulated actions.

\subsection{Language-Conditioned Manipulation}

Language-conditioned manipulation systems connect instructions to actions
through policies, planners, or affordance models. CLIPort~\cite{shridhar2022cliport}
combines language-conditioned semantic features with spatial action maps,
PerAct~\cite{shridhar2022peract} uses voxelized transformer policies,
SayCan~\cite{ahn2022saycan} grounds language-model plans in robot affordances,
and VIMA~\cite{jiang2023vima} studies multimodal prompt-conditioned
manipulation. RT-1~\cite{brohan2022rt1} and RT-2~\cite{brohan2023rt2} scale
robot policies with large datasets and vision-language representations.
VoxPoser~\cite{huang2023voxposer} composes 3D value maps from language and
vision-language models for manipulation planning.
Query-to-Grasp is narrower: it does not learn a manipulation policy, but it
provides a transparent retrieval-to-target layer that such systems often need.

\subsection{RGB-D Grasp Detection and Grasping}

RGB-D grasping methods estimate action targets directly from depth images or
point clouds. Classical and learned grasp detectors include grasp pose
detection in point clouds~\cite{tenpas2017gpd}, synthetic-data grasp planning
with Dex-Net/GQ-CNN~\cite{mahler2017dexnet}, 6-DoF grasp generators such as
Contact-GraspNet~\cite{sundermeyer2021contactgraspnet}, and large grasp
benchmarks such as GraspNet-1Billion~\cite{fang2023graspnet}. AnyGrasp extends
robust grasp perception across spatial and temporal domains~\cite{fang2023anygrasp}.
Query-to-Grasp deliberately does not replace these grasp synthesizers. Instead,
it uses simple scripted executors to isolate how much of the retrieval-to-action
gap is already explained by target-source precision.

\subsection{Active Perception, Multi-View Memory, and Benchmarks}

RGB-D manipulation systems often rely on explicit geometry, object-centric
memory, or multi-view fusion to convert perception into action state. ManiSkill
benchmarks~\cite{gu2023maniskill2} provide controlled simulated environments
for evaluating manipulation pipelines. Query-to-Grasp uses ManiSkill both as
an RGB-D observation source and as the execution substrate for a scripted
top-down pick controller. The key design choice is to keep semantic memory
separate from grasp-target memory so retrieval confidence and downstream
control can be diagnosed independently.

Active perception and next-best-view methods use additional observations to
resolve uncertainty before acting. Query-to-Grasp implements a diagnostic,
rule-based re-observation loop rather than a learned view planner. The policy
requests an extra virtual view when confidence gaps, view support, point
support, or geometry signals are weak. This exposes a useful systems lesson:
closed-loop re-observation can reduce uncertainty metrics without improving
pick success when the extra view is absorbed by the wrong object or shifts the
effective target source.

\section{Method}

\subsection{Single-View Query-to-3D Pipeline}

Given a language query, a deterministic parser extracts a target phrase,
attributes, and conservative relation fields. The perception layer detects
query-conditioned 2D boxes, optionally applies CLIP reranking, and lifts
selected RGB-D pixels into 3D. The lifted semantic center is stored as
\texttt{world\_xyz}. For simulated pick experiments, the system can also compute
an additive grasp target, \texttt{grasp\_world\_xyz}, from local RGB-D geometry.
Semantic retrieval remains unchanged when this grasp point is available.

For a pixel $(u,v)$ with depth $D(u,v)$, intrinsics $K$, and camera-to-world
pose $T_{wc}$, the lifted point is
\begin{equation}
\tilde{\mathbf{p}}_w =
T_{wc}
\begin{bmatrix}
D(u,v)K^{-1}[u,v,1]^\top\\
1
\end{bmatrix}.
\end{equation}
This explicit frame conversion is the boundary between open-vocabulary 2D
retrieval and executable 3D target selection.

The refined grasp point is a deterministic RGB-D target heuristic. Candidate
points are taken only from valid depth pixels inside the selected 2D detection
box, not from all tabletop points. They are then filtered to a table workspace
with
$|x|,|y|\leq0.20$ m and $z\in[-0.02,0.08]$ m; at least 20 workspace points are
required. Let $z_s$ be the 10th percentile of workspace height. Points above
$z_s+0.015$ m form elevated object support; if at least 12 such points exist,
the method clusters their $xy$ coordinates with a 0.03 m radius and selects the
largest low-spread component. The grasp $xy$ is the component median, and the
grasp height is the median local support height within 0.04 m and
$z_s+0.012$ m. If elevated support is insufficient, the system falls back to a
workspace component median and then to the global workspace median. This
produces \texttt{grasp\_world\_xyz}; \texttt{world\_xyz} remains the semantic
center used for retrieval.

\subsection{Multi-View Object Memory}

The multi-view path captures a virtual \texttt{tabletop\_3} preset and adds
per-view detections to object memory. Each memory object stores label votes,
view support, confidence terms, point counts, semantic centers, and optional
grasp observations. Semantic centers support retrieval and target selection.
Grasp observations support downstream simulated pick execution. This separation
allows Table~\ref{tab:target-source} to compare \texttt{selected\_object\_world\_xyz},
\texttt{memory\_grasp\_world\_xyz}, task-guarded semantic targets, and
\texttt{oracle\_object\_pose} without changing the retrieval score.
For compactness, Table~\ref{tab:target-source} abbreviates
\texttt{task\_guard\_selected\_object\_world\_xyz} as TG, MG denotes
\texttt{memory\_grasp\_world\_xyz}, OP denotes
\texttt{oracle\_object\_pose}, and OA/OB denote privileged StackCube cubeA/cubeB
poses.

For memory object $i$, semantic and grasp targets are fused separately:
\begin{equation}
\mathbf{c}_i = \frac{1}{|\mathcal{O}_i|}\sum_{\mathbf{o}\in\mathcal{O}_i}
\mathbf{o}_{xyz}, \qquad
\mathbf{g}_i = \operatorname{median}\{\mathbf{g}_{xyz}:\mathbf{g}_{xyz}
\text{ valid}\}.
\end{equation}
The first term preserves retrieval geometry; the second supplies the action
target when refined simulated picking is enabled.

An observation is merged into the nearest compatible memory object within
0.08 m; otherwise a new hypothesis is created. Compatibility requires a
matching normalized label or a parsed-attribute match for the target phrase;
objects with conflicting color or class votes are not merged even if they are
nearby. Label votes, detector scores, CLIP scores, view ids, point counts, and
grasp observations are accumulated.
Overall confidence is a clipped weighted average of detector score (0.30),
CLIP score (0.30), view support (0.15), spatial consistency (0.15), and
geometry support (0.10). Thus the same memory object can expose both a semantic
center for retrieval and a median grasp point for execution.

\subsection{Re-Observation Policy}

The re-observation policy is rule-based and diagnostic. It triggers if the
selected object has fewer than two supporting views, confidence gap below
0.05, overall confidence below 0.50, geometry confidence below 0.50, or mean
3D point support below 100. Suggested additional views are \texttt{top\_down}
and \texttt{closer\_oblique}. A closed-loop benchmark can capture one extra
view, merge it into memory, and rerun target selection.
Continuity guards keep selected-object association stable when appropriate,
while absorber-aware checks prevent forcing continuity when a competing object
also absorbed the extra view.

\subsection{Simulated Pick and Pick-Place Execution}

The retrieval-only baseline returns a structured non-execution result. The
\texttt{sim\_topdown} executor uses ManiSkill \texttt{pd\_ee\_delta\_pos}
actions to move the TCP 0.12 m above the target, descend to target height plus
0.025 m, close the gripper, and lift to target height plus 0.18 m. The stage
budgets are 40/30/20/40 simulator steps for move, descend, close, and lift.
The \texttt{sim\_pick\_place} executor extends the sequence with 60
move-to-place steps, 30 place-descend steps, 20 open-gripper steps, and 20
settle steps, releasing at the place target plus 0.06 m. Both executors report
\texttt{grasp\_attempted}, \texttt{pick\_success}, \texttt{place\_success},
\texttt{task\_success}, stage diagnostics, final TCP position, and simulator
info. \texttt{pick\_success} measures task-specific grasp or lift evidence.
For pick-place rows, \texttt{place\_success} is confirmed by ManiSkill
placement evidence when available, falling back to the environment task flag;
\texttt{task\_success} is the raw ManiSkill task success flag. For pick-only
rows, the raw task flag is recorded but is not the primary metric.

The scripted executors are a deliberate diagnostic choice. A learned controller
could compensate for some target errors, but it would also entangle perception
quality with policy adaptation. By using fixed simulated controllers, the
experiments ask a narrower question: whether the selected target source is
already good enough to be executed by a transparent pick or pick-place routine.

\begin{algorithm}[t]
\caption{Query-to-Grasp Target Selection and Execution}
\label{alg:q2g}
\begin{algorithmic}[1]
\Require query $q$, RGB-D views $\mathcal{V}$, executor mode $e$
\State parse $q$ into grounding prompt and optional reference query
\For{view $v \in \mathcal{V}$}
  \State detect boxes with GroundingDINO and optionally rerank with CLIP
  \State lift valid depth pixels to \texttt{world\_xyz}
  \State compute optional \texttt{grasp\_world\_xyz} from workspace support
  \State add semantic and grasp observations to object memory
\EndFor
\State select target source: semantic center, memory grasp, task guard, or oracle
\If{re-observation thresholds trigger}
  \State capture suggested diagnostic view and repeat detection/fusion
\EndIf
\If{$e$ is \texttt{sim\_topdown}}
  \State execute scripted pick from selected target
\ElsIf{$e$ is \texttt{sim\_pick\_place}}
  \State select oracle or predicted reference-object place target
  \State execute scripted pick-place and record pick/place/task metrics
\Else
  \State return retrieval-only diagnostic artifact
\EndIf
\end{algorithmic}
\end{algorithm}

\section{Experimental Setup}

Experiments run in ManiSkill with RGB-D observations and the high-fidelity
GroundingDINO backend. The primary benchmark is \texttt{PickCube-v1}.
\texttt{StackCube-v1} is the cross-task bridge benchmark: query-driven rows
test cubeA picking, oracle rows test privileged cubeA-to-cubeB placement, and
the query-pick/oracle-place bridge tests query-derived cubeA targets with an
oracle cubeB placement target. The final paper-revision runs also test a
non-oracle explicit reference-object bridge in which the pick query is
\texttt{red cube} and the place-reference query is \texttt{green cube}.

The experimental protocol is summarized in Table~\ref{tab:protocol}. PickCube
is used to test whether query-conditioned 3D targets can become executable
grasp points. StackCube is used as a stricter cross-task diagnostic: the
pick-only rows measure cubeA graspability, the oracle rows measure controller
and placement feasibility, and the query-pick/oracle-place bridge measures how
far query-derived pick targets can support task completion when the destination
is privileged.

\texttt{grasp\_attempted\_rate}, \texttt{pick\_success\_rate},
\texttt{place\_success\_rate}, \texttt{task\_success\_rate}, and stage counts.

The experiments answer four diagnostic questions. Does language reranking
change the selected target, or is 3D target quality the limiting factor? Does
camera-frame alignment make multi-view memory meaningful? Can a selected
memory target drive an executable pick on the primary task? And, when moving to
StackCube, are failures caused by the controller alone or by target-source and
association choices before execution? The oracle rows in Table~\ref{tab:target-source}
are therefore not performance claims for a deployable system; they are
controlled probes for separating perception, target selection, and placement
feasibility.

\section{Results}

\subsection{Grounding and Reranking Baseline}

The single-view baseline establishes that the language-to-3D chain is stable
before adding multi-view memory. Exact PickCube queries produce a 3D target
across the validated settings. Ambiguity queries increase candidate counts, but
CLIP reranking does not emerge as the primary improvement source: top-1 changes
are absent because candidate multiplicity is low. In the paper-revision
ablation, PickCube with CLIP reaches \texttt{50/50} pick success with
\texttt{0/50} top-1 changes and mean detection count \texttt{1.04}; StackCube
with CLIP reaches \texttt{44/50} with \texttt{0/50} top-1 changes and mean
detection count \texttt{1.48}. This motivates the rest of the paper, which
focuses on 3D action-target quality rather than semantic reranking.
This result is important because it prevents the paper from over-attributing
execution failures to semantic ranking. Under the evaluated proposal sets, the
more consequential error is whether the selected 2D evidence becomes a stable
and graspable 3D target.

\subsection{Camera Geometry Enables Memory}

The OpenCV-to-OpenGL camera-convention correction is the first critical
systems result. After the correction, same-label cross-view spread drops from
\texttt{1.0693 m} to \texttt{0.0518 m}, and memory fragmentation drops from
\texttt{3.3333} to \texttt{1.3333} objects per run. Fig.~\ref{fig:geometry}
visualizes this change.

\subsection{Fused Grasp Points Enable PickCube Picking}

Semantic fused centers are useful for retrieval, but they are not always good
pick targets. Propagating per-view grasp points into fused memory and using
\texttt{memory\_grasp\_world\_xyz} for refined multi-view simulated picking
raises compact and full-query PickCube pick success to \texttt{1.0000}. The
target-point ablation makes this concrete: semantic centers reach
\texttt{53/55} = \texttt{0.9636}, while refined grasp points reach
\texttt{55/55} = \texttt{1.0000}. This is the strongest positive result: the
retrieval target, fused memory, and controller can agree well enough to execute
successful simulated picks across the full ambiguity set.

\subsection{StackCube Exposes Cross-Task Limits}

StackCube validates that the pipeline can transfer to a second ManiSkill task
for pick-only control, but it also exposes task-specific target-source
preferences. A StackCube guard uses the selected object's semantic fused center
as the effective refined multi-view target, yielding \texttt{0.6200} tabletop
(\texttt{31/50}) and \texttt{0.5200} closed-loop (\texttt{26/50}) pick success
over 50 seeds.

The non-oracle predicted-place bridge removes the privileged destination while
keeping the language interface explicit: the pick query is \texttt{red cube}
and the reference-place query is \texttt{green cube}. Over 500 no-CLIP seeds,
the refined predicted-place bridge reaches \texttt{0.5520} task success in
single-view (\texttt{276/500}), \texttt{0.4720} in tabletop multi-view
(\texttt{236/500}), and \texttt{0.4460} in closed-loop (\texttt{223/500}).
The broad reference query \texttt{cube} is substantially weaker, reaching only
\texttt{0.3600} task success in single-view over 200 seeds. This does not make
StackCube a solved language-conditioned stacking task; instead it shows that
reference-object specificity is itself a measurable target-source variable.

Privileged placement rows are reported as diagnostic upper bounds. With the
destination fixed to \texttt{oracle\_cubeB\_pose}, query-derived cubeA pick
targets reach \texttt{0.7200} task success in the single-view setting
(\texttt{36/50}), \texttt{0.5200} in tabletop multi-view (\texttt{26/50}), and
\texttt{0.4800} in closed-loop multi-view (\texttt{24/50}). A fully oracle
pick-place probe reaches \texttt{0.8800} task success (\texttt{44/50}), and an
oracle-pick probe reaches \texttt{0.9400} pick success (\texttt{47/50}). These
rows do not claim deployable StackCube completion; they show how much of the
remaining task-success gap is tied to target-source quality and association
before placement.

Table~\ref{tab:noisy-oracle} explains why the previous multi-view result is
diagnostic rather than paradoxical. Multi-view fusion can increase target
availability and attempted actions, but if aggregation shifts the effective
action target by a few centimeters, the controller becomes less likely to
succeed. Thus, the paper's main claim is not that more views are always better;
it is that the evaluation can reveal the recall--precision tradeoff at the
action-target level. This sensitivity also explains why visually small
cross-view or reference-target errors can dominate downstream task success.

\subsection{Closed-Loop Re-Observation Is Diagnostic, Not Always Beneficial}

Closed-loop re-observation reduces uncertainty diagnostics on PickCube and in
the accepted absorber-aware policy path, but it does not improve StackCube pick
success. The expanded StackCube failure report attributes failures to wrong
fused grasp observations, memory fragmentation or low support, controller or
contact residuals, and closed-loop third-object absorption. This supports a
conservative interpretation: re-observation is useful for diagnosis and can
resolve selection uncertainty, but it is not automatically a grasp-success
improvement. Fig.~\ref{fig:stackcube-failures} summarizes the expanded
50-seed failure taxonomy.
This distinction is central to the retrieval-to-execution thesis. A view that
reduces semantic uncertainty can still perturb object association or move the
effective action target toward a distractor. The failure taxonomy therefore
argues against treating active perception metrics and manipulation success as
interchangeable objectives.

\subsection{Discussion}

Taken together, the experiments identify target-source quality as the central
bottleneck between open-vocabulary RGB-D retrieval and executable manipulation.
CLIP-style semantic reranking is useful infrastructure, but it is not the
dominant lever in these candidate pools. Geometry correction and memory-grasp
fusion solve the primary PickCube execution path, while StackCube shows that
task-specific target choice, object association, and placement-source
privilege determine how far the same retrieval stack transfers. The external
crop baseline adds a reviewer-facing comparison: a selected 2D box is not
enough if the 3D point source is a noisy single center pixel, while crop-level
aggregation and top-surface filtering largely close the PickCube pick gap.
Additional target-source formation scans show that the perception-to-3D-target
interface also runs on PushCube, LiftPeg, PegInsertion, and StackPyramid
variants (\texttt{1.000}, \texttt{1.000}, \texttt{1.000}, and \texttt{0.970}
target formation rates over 200 seeds, respectively), but those rows are not
manipulation-success claims because the scripted top-down controller is not
task-compatible with all of these families. The 50-seed non-cube gate further
separates compatibility from execution: PickSingleYCB, PickClutterYCB, and
PickSingleEGAD fail at runtime in the installed simulation setup and are
therefore not counted as negative manipulation evidence. LiftPeg is a valid
executor-mismatch probe: \texttt{oracle\_object\_pose} and
\texttt{crop\_top\_surface} provide targets for all 50/50 runs, but
\texttt{sim\_topdown} still reaches \texttt{0.000} pick success. This means
target-source quality is necessary but not sufficient when object geometry and
controller approach are mismatched. This is why the paper treats oracle
targets, noisy-oracle probes, query/oracle bridges, and non-cube diagnostics as
instruments: they expose which part of the retrieval-to-execution chain fails
before adding more policy capacity.

\section{Limitations}

Query-to-Grasp is intentionally a diagnostic simulated systems study. It does
not report physical robot experiments, a learned controller, or full long-horizon
manipulation. These are not merely missing components; they are held fixed so
that target-source quality, cross-view consistency, and re-observation
association can be measured without policy learning as a confound. The
\texttt{sim\_topdown} executor is a scripted top-down pick policy, and
\texttt{sim\_pick\_place} is a scripted placement probe rather than a learned
controller.

StackCube is also deliberately separated into claim levels. Pick-only rows
measure whether query-derived targets can support grasping. Query-pick/oracle
place rows use a privileged destination target and therefore serve as partial
task-success bridges, not deployable language-conditioned stacking. Fully
oracle pick-place rows are upper bounds that diagnose controller and placement
feasibility. Predicted-place rows remove the oracle destination, but they still
use an explicit reference-object query and a scripted controller; they should be
read as a non-oracle diagnostic bridge, not as robust relation-heavy stacking.

The detector backend is used off the shelf. The current results do not show
CLIP reranking as a primary improvement source under the evaluated candidate
pools. The query parser handles simple attributes and conservative relation
fields, not unrestricted relation-heavy language reasoning. Multi-view
re-observation is implemented with virtual views in ManiSkill, not physical
camera motion. These boundaries leave clear next steps: stronger non-oracle
relation grounding, more diverse task-compatible controllers, learned or
adaptive policies as separate ablations, and eventual real-robot validation.

\section{Conclusion}

Query-to-Grasp connects language-queryable RGB-D perception to multi-view 3D
memory, uncertainty-aware re-observation, and opt-in simulated pick and
pick-place execution.
The current system shows that careful camera geometry and separate grasp-point
memory are enough to turn PickCube from a retrieval-only benchmark into a
strong simulated pick benchmark. StackCube demonstrates useful cross-task
transfer while exposing target-source, placement-target, reference-query, and
closed-loop association limits. Noisy-oracle perturbations show that
centimeter-scale target errors are enough to explain large execution losses,
and the 500-seed predicted-place bridge shows that explicit reference-object
perception can partially close the oracle-placement gap. External crop-grasp
baselines show that box-center depth is a poor manipulation target, while crop
aggregation and top-surface selection are strong non-oracle baselines in clean
cube scenes. A non-cube gate shows that YCB/EGAD variants require compatible
runtime assets and that LiftPeg still fails even with oracle targets when the
executor is geometrically mismatched. These results provide a reproducible
basis for a simulated diagnostic systems paper. The next
technical steps are richer relation grounding, broader task-compatible
controllers, and real-world transfer, each of which should be evaluated as a
separate source of evidence rather than folded into the present target-source
diagnosis.
